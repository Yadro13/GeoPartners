# Резервні копії та відновлення

## Політика

- Сервіс `geopartners-backup` запускається щонеділі о 02:00 UTC.
- PostgreSQL зберігається у custom-архіві `pg_dump`.
- Перед завантаженням архів перевіряється командою `pg_restore --list` і наявністю ключових таблиць.
- Архів завантажується у Railway Object Storage до `backups/database/`.
- Об'єкти старші за 90 днів видаляються після успішного створення нового архіву.
- PDF ділянок зберігаються окремо під префіксом `plots/` і не підпадають під 90-денне очищення резервних копій.

## Перевірка архіву

Перевірити найновіший архів у Bucket:

```bash
npm run backup:verify
```

Перевірити конкретний об'єкт або локальний файл:

```bash
npm run backup:verify -- --key backups/database/geopartners-TIMESTAMP.dump
npm run backup:verify -- --file /path/to/geopartners.dump
```

Для Object Storage потрібні `AWS_*` змінні. У середовищі також має бути `pg_restore`.

## Тестове відновлення

Відновлення завжди виконується в окрему порожню БД. Скрипт відмовиться працювати, якщо ціль збігається з `DATABASE_URL`.

1. Створіть тимчасову PostgreSQL БД, наприклад `geopartners_restore_drill`.
2. Встановіть `RESTORE_DATABASE_URL` на цю БД.
3. Встановіть `RESTORE_CONFIRM_DATABASE=geopartners_restore_drill`.
4. Запустіть відновлення:

```bash
npm run backup:restore -- --file /path/to/geopartners.dump
```

Або без локального завантаження:

```bash
npm run backup:restore -- --key backups/database/geopartners-TIMESTAMP.dump
```

Після `pg_restore` скрипт підключається до цільової БД і перевіряє наявність ключових таблиць. Для повного квартального drill додатково перевірте кількість користувачів, ділянок в обох областях, останні записи журналу та формування звіту.

Якщо користувач PostgreSQL має право створювати окремі бази, повну технічну перевірку можна виконати однією командою:

```bash
npm run backup:drill
```

Команда автоматично вибирає найновіший архів, створює тимчасову БД, виконує відновлення та перевірку, а потім видаляє тимчасову БД навіть у разі помилки.

## Аварійне відновлення робочої БД

1. Зупиніть запис у застосунок або тимчасово вимкніть web-сервіс.
2. Створіть нову порожню PostgreSQL БД.
3. Відновіть потрібний архів у нову БД за процедурою вище.
4. Перевірте дані через тимчасове підключення web-сервісу.
5. Змініть `DATABASE_URL` web, notifications і backup на перевірену БД.
6. Запустіть міграції, web і worker; перевірте `/api/health`.
7. Стару БД не видаляйте до завершення приймальної перевірки.

Пряме відновлення поверх активної робочої БД не використовується.

## Періодичність перевірок

- Щотижня: статус cron та наявність нового об'єкта.
- Щомісяця: `backup:verify` для найновішого архіву.
- Щокварталу: повне відновлення в окрему БД і функціональна перевірка.
