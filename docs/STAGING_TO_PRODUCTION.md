# Dry-run перенесення staging у production

Ця процедура готує контрольоване перенесення перевірених даних GeoPartners з Railway staging у production. До рішення замовника вона використовується лише як план і для репетиції на тимчасових ресурсах. Чинний production не змінюється.

## Що переносимо

- PostgreSQL: користувачі, реєстрації, налаштування, робоча й тестова області, категорії, ділянки, аудит, версії та notification outbox.
- Railway Bucket: PDF ділянок і архіви БД, якщо для production не створено окрему політику архівів.
- Конфігурацію застосунку не копіюємо як дані. Railway variables, домен, OAuth callback, SMTP і Telegram перевіряються окремо.

## Передумови

- Зафіксовано release commit у `staging`, CI і staging smoke test зелені.
- Замовник підтвердив фінальні правила звіту, домен і дату перенесення.
- Створено порожні production PostgreSQL і Bucket з окремими credentials.
- Відоме коротке вікно зупинки запису. На час фінального backup зупиняються production/staging web і notification worker, які можуть змінювати джерело.
- Призначено відповідального за рішення `go/no-go` і перевірено шлях rollback.

## Репетиція без production

1. Запустити актуальний staging backup:

   ```powershell
   npm run backup:database
   npm run backup:verify
   ```

2. Відновити останній архів тільки у тимчасову БД:

   ```powershell
   npm run backup:drill
   ```

3. Перевірити, що архів містить усі критичні таблиці: auth, registration, notification outbox, settings, plots, audit і versions.
4. Побудувати інвентар Bucket без виведення object keys у спільні журнали: кількість PDF, кількість backup-архівів і загальний обсяг.
5. Скопіювати об'єкти у тимчасовий Bucket через S3 API з режимом без перезапису.
6. Порівняти джерело і тимчасову ціль за кількістю об'єктів, сумарним розміром та checksum/ETag там, де він придатний.
7. Підняти тимчасовий web на відновленій БД і Bucket та пройти реальний API/E2E чек-лист.
8. Видалити тимчасові ресурси тільки після збереження результату репетиції у приймальному чек-листі.

## Фінальне перенесення

1. Зафіксувати commit SHA, час початку і стан health-check.
2. Увімкнути вікно без запису та дочекатися завершення активних імпортів.
3. Запустити фінальний backup, `backup:verify` і записати розмір та час архіву.
4. Відновити PostgreSQL у порожню production БД через `scripts/restore-database.mjs`. Скрипт має отримати `RESTORE_DATABASE_URL` і точне підтвердження `RESTORE_CONFIRM_DATABASE`; відновлення поверх `DATABASE_URL` заблоковано.
5. Скопіювати Bucket зі staging у production без видалення джерела.
6. Звірити контрольні показники:
   - кількість користувачів і заявок за статусами;
   - кількість ділянок у `production` і `sandbox`;
   - кількість категорій, audit entries та plot versions;
   - pending/failed/exhausted notification outbox;
   - кількість PDF та відсутність посилань на відсутні об'єкти.
7. Підключити production web/worker/backup до нових PostgreSQL і Bucket.
8. Перевірити `/api/health`, вхід адміністратора, права ролей, карту, відкриття PDF, імпорт, журнал, чергу сповіщень і формування поточного звіту.
9. Тільки після `go` переключити домен/DNS і Google OAuth callback.

## Rollback

- Не видаляти staging PostgreSQL, Bucket і фінальний backup щонайменше до завершення приймання.
- До перемикання DNS rollback означає повернення Railway variables на попередні production ресурси.
- Після перемикання DNS повернути попередній deployment і ресурси, потім повторно перевірити health/auth.
- Не запускати двосторонню синхронізацію. Після початку вікна перенесення джерелом істини залишається зафіксований staging snapshot.

## Запис результату

Для кожної репетиції або перенесення фіксуються дата, commit SHA, backup timestamp, кількість записів/об'єктів, результат E2E, відповідальний і рішення `go/no-go`. Значення credentials, адресати сповіщень, object keys і персональні дані у протокол не вносяться.
