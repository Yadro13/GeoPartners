# Інтеграційний тест Railway staging

Сценарій `npm run test:staging` перевіряє зібраний GeoPartners проти реальної PostgreSQL у Railway staging.

## Що перевіряється

- реєстрація адміністратора та користувача через штатний Better Auth API;
- доставка і проходження посилання підтвердження email через Mailpit;
- створення заявки, повідомлення адміністратору, схвалення та коментар у листі користувачу;
- сесії, очікування схвалення, призупинення доступу і захист головного адміністратора;
- права `user` та `admin` для CRUD ділянок, імпорту, категорій, аудиту і відновлення;
- ізоляція робочої та тестової областей;
- видалення тимчасових користувачів, сесій, заявок, ділянок, аудиту, версій, outbox і листів.

## Ізоляція

Runner можна виконати лише коли Railway передає одночасно:

```text
RAILWAY_ENVIRONMENT_NAME=staging
RAILWAY_SERVICE_NAME=geopartners-web
```

Він піднімає другий екземпляр уже зібраного Next.js на `127.0.0.1:3100`. Дочірній процес використовує ту саму staging PostgreSQL, але окремі одноразові адреси `example.invalid`, локальний Mailpit SMTP та вимкнений Telegram. Brevo, реальна адреса адміністратора і production не залучаються.

Усі записи мають унікальний ідентифікатор запуску. Блок `finally` виконує адресне очищення навіть після помилки та перевіряє відсутність залишків у ключових таблицях.

## Запуск

Запускати команду потрібно всередині активного Railway staging web-контейнера:

```bash
npm run test:staging
```

Успішний результат завершується рядком:

```text
[staging-e2e] PASS
```

Mailpit використовує приватні адреси `mailpit.railway.internal:1025` для SMTP і `mailpit.railway.internal:8025` для API. Його UI та API bind мають залишатися `MP_UI_BIND_ADDR=[::]:8025`, SMTP bind — `MP_SMTP_BIND_ADDR=[::]:1025`, щоб сервіс працював у Railway private networking.
